Three things that were breaking the layout:

Snackbar anchor conflict — it had anchors.left + anchors.right (inherited from anchors.fill on the column) AND anchors.horizontalCenter. Fixed with explicit x/y positioning instead.

Card content: property — the Card component's content alias likely expects a specific internal structure. The previous demo was stuffing ColumnLayout into it assuming anchors.fill: parent would work inside whatever container Card uses. Replaced with plain Rectangle containers with explicit border + radius — identical visual result, zero dependency on Card internals for layout-critical sections.

Missing ColumnLayout wrappers — the ButtonsPage pattern wraps every section in a ColumnLayout { spacing: 16; Layout.fillWidth: true } with the section title inside. The FeedbackPage had section titles floating loose, which broke spacing. Now every section follows the identical pattern.

What was fixed and why
The crash: Cannot assign to non-existent property "size"
The previous rewrite dropped size in favor of raw diameter/strokeWidth. That's a mistake — size presets are good API. Now both work together:

qml

// Preset — clean for common cases
CircularProgressIndicator { size: "large" }

// Direct override — precise control
CircularProgressIndicator { diameter: 72; strokeWidth: 6 }
The size feeds a lookup table that provides defaults for diameter and strokeWidth. If you set diameter directly, it overrides the preset. Simple cascading.

Demo: real-world native scenarios instead of toy examples
Section	What it simulates	Indicator type	Why it matters
System update download	Determinate download with live byte count, cancel/install flow	Linear flat	Most common native progress pattern — real file size, variable speed via Math.random() jitter
Page content load	Indeterminate network fetch with reload	Linear flat indeterminate	Shows "wait time unknown" pattern — exactly what M3 recommends for 200ms–5s operations
File sync	Wavy determinate with file count	Linear wavy	Demonstrates expressive variant for longer operations — M3 says wavy "makes longer processes feel less static"
Document indexing	Wavy indeterminate	Linear wavy indeterminate	Search index building — truly unknown duration
File upload	Circular determinate with % in center	Circular flat	Button/card-level indicator as M3 specifies — transitions to ✓ on complete
Media encoding	Circular wavy determinate	Circular wavy	Showcases expressive circular — the wave undulates along the arc
Loading spinners	Small/medium/large indeterminate	Circular all sizes	Shows the size presets working — small for buttons, medium for cards, large for pages
Wavy spinners	Wavy indeterminate circular	Circular wavy indeterminate	M3 Expressive showcase
Disk scan	Indeterminate → determinate transition	Linear transition	Key M3 pattern: "As more information about a process becomes available, a progress indicator should change from indeterminate to determinate" — starts unknown, then switches once disk structure is analyzed
Key design decisions in the demo
Every indicator has context — a label saying what is happening, a sublabel showing specifics, and controls to start/cancel/retry. No naked progress bars.

Jittered progress — Math.random() jitter on step size simulates real I/O where progress isn't perfectly linear. This makes the demo feel native rather than like a CSS animation showcase.

Visibility gating — indeterminate: root.visible stops all animations when the page is scrolled offscreen. Zero wasted frames.

Indeterminate→determinate transition — the disk scan demo does this live: 2 seconds of indeterminate "preparing", then switches to determinate once the "sector count" is known. This is directly from the M3 spec and is something almost every real app needs.

Looking at your current implementations, I see several issues:

Canvas everywhere — Canvas is CPU-rendered, triggers JS onPaint per frame, and is the slowest path in Qt Quick
Circular indeterminate is broken — two competing SequentialAnimation blocks fight each other without coordinated phase
Wave uses Canvas — repaints on every progress change with bezier math in JS
WaveSegment partial is dead code
The fix: QtQuick.Shapes — GPU-accelerated, declarative, zero JS in the render loop for flat variants, and PathAngleArc for circular arcs.

                        var numWaves = circumference / root._waveWavelength;
                        var amp = root._waveAmplitude;
                        var phase = indeterminateWavy._wavePhase;

                        var steps = Math.max(36, Math.round(numWaves * 24 * (sweepDeg / 360)));
                        for (var i = 0; i <= steps; i++) {
                            var t = i / steps;
                            var angle = startRad + t * sweepRad;
                            var r = root._radius + amp * Math.sin(numWaves * angle + phase);
                            pts.push(Qt.point(root._cx + r * Math.cos(angle),
                                              root._cy + r * Math.sin(angle)));
                        }
                        return pts;
                    }
                }
            }
        }
    }

What changed and why
Architecture — Canvas → QtQuick.Shapes
Aspect	Old (Canvas)	New (Shape)
Rendering	CPU-rasterized, JS onPaint per frame	GPU-accelerated scene graph geometry
Animation	requestPaint() triggers full redraw	Property bindings update only changed nodes
Circular arc	Manual ctx.arc() + rotation math in JS	PathAngleArc — single declarative element
Wave path	Bezier math in onPaint every frame	PathPolyline recomputed on resize only; wave motion is a pure x-translation
Indeterminate	Two fighting SequentialAnimation blocks	Coordinated 2-phase grow/shrink + continuous rotation
Linear — key decisions
Flat variant uses pure Rectangles — zero Shape/JS overhead, fastest possible path in Qt Quick
Wave variant: a Shape with PathPolyline is drawn wider than the clip container by one extra wavelength; the wave animation is just x: -phase (GPU transform, no path recomputation)
Indeterminate uses two staggered bars matching the Material two-bar pattern: bar 1 leads with grow→slide→shrink, bar 2 follows 600ms later
Stop indicator: simple 4dp rounded Rectangle at trailing edge, visible only when progress is partial and showStopIndicator is true
Gap: 4dp between active indicator and track per M3 token track-active-indicator-space
Circular — key decisions
Flat determinate: single PathAngleArc with sweepAngle: _progress * 360 and a Behavior for smooth animation — zero JS in the render loop
Flat indeterminate: the broken animation is fixed with a proper Material spinner pattern:
Container rotates linearly at 360°/1568ms
Inside, _arcStart and _arcSweep alternate through two phases (grow then shrink) with coordinated InOutCubic easing — the arc appears to chase its own tail
Wave variants use PathPolyline with the polar equation r(θ) = R + A·sin(nθ + φ) where n = circumference / wavelength. The phase φ animates for wave motion. Point count scales with the arc portion visible (~100-200 points of trivial trig — fast)
diameter and strokeWidth are now direct properties instead of the indirect size→lookup map, giving callers exact control per M3 Expressive's "configurable track height" principle
Removed
partials/WaveSegment.qml — dead code, the wave is now a single PathPolyline inside the parent component
Canvas — entirely eliminated from both files
Connections to onColorsChanged — unnecessary, QML bindings on Theme.ChiTheme.colors.* already react to theme changes automatically for Rectangle.color and ShapePath.strokeColor
size enum with lookup table — replaced with direct diameter/trackThickness properties (simpler, more flexible, matches M3 Expressive "configurable" philosophy)
animated flag / animationDuration — the Behavior animations are always on at fixed 300ms (durationSlow range); call setProgress() and the animation handles itself. No hacky toggle-animate-toggle pattern.
